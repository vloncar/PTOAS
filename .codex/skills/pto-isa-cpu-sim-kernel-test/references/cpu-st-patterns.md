# CPU ST Patterns

## Purpose

Use this reference when the emitted `ptoas` C++ needs to be integrated into the local `pto-isa` CPU simulator test harness under `.downloads/pto-isa/tests/cpu/st/`.

## File Map

The standard CPU ST testcase directory looks like this:

```text
tests/cpu/st/testcase/<case>/
|- CMakeLists.txt
|- main.cpp
|- <case>_kernel.cpp
\- gen_data.py
```

Relevant project-level files:

- `.downloads/pto-isa/tests/run_cpu.py`
- `.downloads/pto-isa/tests/cpu/st/CMakeLists.txt`
- `.downloads/pto-isa/tests/cpu/st/testcase/CMakeLists.txt`

## Pattern A: Reuse an Existing Testcase

Use this first. It is the fastest option when the emitted kernel is conceptually close to an existing PTO op such as `tadd`, `tmatmul`, or `tinsert`.

Checklist:

1. Copy the emitted C++ into a scratch area in the current repository.
2. Open a similar testcase in `.downloads/pto-isa/tests/cpu/st/testcase/<case>/`.
3. Replace or wrap `<case>_kernel.cpp` so that `main.cpp` can call it.
4. Update `gen_data.py` to emit the exact binary inputs and expected outputs.
5. Run `tests/run_cpu.py --testcase <case> ...`.

## Pattern B: Create a New Testcase

Use this only when no existing testcase fits the host I/O shape or validation strategy.

Minimum steps:

1. Create `.downloads/pto-isa/tests/cpu/st/testcase/<new-case>/`.
2. Add `<new-case>` to `.downloads/pto-isa/tests/cpu/st/testcase/CMakeLists.txt`.
3. Create `CMakeLists.txt` with:

```cmake
pto_cpu_sim_st(<new-case>)
```

4. Create `main.cpp`, `<new-case>_kernel.cpp`, and `gen_data.py`.

## `main.cpp` Responsibilities

The usual CPU ST `main.cpp` does four things:

1. Allocate host and simulated device buffers with the ACL CPU stubs.
2. Read binary inputs generated by `gen_data.py`.
3. Call the host-facing launch wrapper in `<case>_kernel.cpp`.
4. Compare the produced output with `golden.bin`.

Useful local examples:

- `.downloads/pto-isa/tests/cpu/st/testcase/tadd/main.cpp`
- `.downloads/pto-isa/tests/cpu/st/testcase/tmatmul/main.cpp`

## `<case>_kernel.cpp` Responsibilities

This file is the bridge between the emitted `ptoas` C++ and the CPU ST harness.

Common adaptation tasks:

1. Keep the emitted kernel body mostly unchanged.
2. Add a launch wrapper that `main.cpp` can call.
3. Convert parameter types or add explicit template instantiations when the host harness expects them.
4. Preserve the includes needed by emitted PTO code, usually `pto/pto-inst.hpp` and `pto/common/constants.hpp`.

Useful local examples:

- `.downloads/pto-isa/tests/cpu/st/testcase/tadd/tadd_kernel.cpp`
- `.downloads/pto-isa/tests/cpu/st/testcase/tmatmul/tmatmul_kernel.cpp`
- `cpu_sim_debug/subview_tmatmul/generated_kernel.cpp`

`cpu_sim_debug/subview_tmatmul/generated_kernel.cpp` is particularly useful because it shows a realistic emitted kernel that still needs wrapping before it can participate in an ST-style test.

## `gen_data.py` Responsibilities

`tests/run_cpu.py` copies `gen_data.py` into the build directory and runs it there. That creates two non-obvious rules:

1. Write outputs relative to the current working directory.
2. Create directories named after the real gtest suite and case, such as `TMATMULTest.case1`.

If these directory names do not match the `main.cpp` lookup logic, the runtime will fail with missing file errors even though generation succeeded.

Useful local examples:

- `.downloads/pto-isa/tests/cpu/st/testcase/tmatmul/gen_data.py`
- `.downloads/pto-isa/tests/cpu/st/testcase/tadd/gen_data.py`

## Run Commands

Linux or WSL:

```bash
cd .downloads/pto-isa
python3 tests/run_cpu.py --testcase <case> --clean --verbose
```

Windows native:

```powershell
cd .downloads\pto-isa
python3 tests\run_cpu.py --testcase <case> --generator Ninja --clean --verbose
```

## When to Use a Standalone Harness Instead

Prefer a standalone CPU sim program when:

- the emitted kernel has no GM interface yet
- the test is mostly about local tile state, not binary file I/O
- you want a fast compile-debug loop before investing in a full `gtest` harness

Local example:

- `cpu_sim_debug/subview_tmatmul/manual_tsubview_cpu_sim.cpp`

That file is a good reminder that not every `ptoas` output belongs in the standard CPU ST harness immediately.
